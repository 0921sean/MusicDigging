## Problem Statement

캐주얼 리스너(20대, 특정 장르 없이 좋은 노래 반복재생하는 유형)는 새 노래를 발견하고 싶지만
능동적으로 탐색할 도구가 없다. YouTube Music 추천은 수동적(자동재생 중 우연히 걸림)이고,
친구 추천에 의존하면 불규칙하다. 결과적으로 좋은 노래 발견 → 반복재생 → 더 파고 싶은데 막힘의
사이클이 반복된다. "전문 디깅 사이트"는 존재하지만 캐주얼 리스너가 접근할 수 없다.

## Demand Evidence

- 본인이 첫 번째 유저. 친구가 추천해준 완전 새로운 장르 노래를 듣고 좋았지만,
  그 장르를 더 파는 방법을 몰라 다시 반복재생으로 돌아감.
- "이 앱이 사라지면 음악적 깊이가 다시 갇힐 것 같다"는 표현 — 직접적 수요 근거.
- 한 달 최대 10곡 추가가 "역대급"이라는 사실이 탐색 빈도가 얼마나 낮은지 보여줌.

## Status Quo (현재 대안)

1. **친구 추천** — 수동적, 불규칙, 확장 불가
2. **YouTube Music 자동재생** — 컴퓨터 틀어놓다 우연히 발견, 의도적 탐색 아님

"능동적으로 탐색하려고 앱을 켜는" 행동 자체가 없음.

## Target User & Narrowest Wedge

**타겟:** 20대, 장르 불문하고 좋은 노래 반복재생하는 캐주얼 리스너.
하루 대부분(이동, 작업, 운동)에 음악 틀어놓지만 한 달에 새 노래 10곡 이하 추가.

**Narrowest Wedge:**
좋아하는 노래 5곡 입력 → 비슷한 느낌의 새 노래 추천 → 스와이프(좋다/싫다)로 취향 신호 전달.
이 루프 하나만. 플레이리스트 import 없음, 스트리밍 연동 없음.

## Constraints

- Day 1: 웹앱 (모바일 브라우저에서 동작하면 충분)
- 스트리밍 연동(Spotify/YouTube/Melon 딥링크)은 나중에 추가
- 추천 엔진 직접 구현 X — Spotify Recommendations API 또는 Last.fm Similar Tracks API 사용
- 장르이탈 버튼: 메인 추천 모델에 영향 없이 다른 endpoint 호출 (별도 상태 유지)

## Premises (확정)

1. 타겟은 음악 덕후가 아닌 캐주얼 리스너. 복잡한 온보딩이나 전문 용어 없어야 함.
2. 현재 대안(YouTube 자동재생, 친구 추천)은 둘 다 수동적이고 불안정.
3. 핵심 차별점은 알고리즘 품질이 아니라 "탐색을 하나의 의도적 활동으로 만드는 UX".
4. 스트리밍 연동은 MVP에서 제외. 추천 → 평가 루프 검증이 먼저.

## Approaches Considered

### Approach A: "오늘 배포" (선택됨)
씨드 트랙 직접 입력 → Last.fm / Spotify API 추천 → 스와이프 UI.
Effort: S. Risk: Low. 이번 주 안에 본인이 실제로 써볼 수 있음.

### Approach B: "제대로 된 앱"
플레이리스트 import + OAuth + 취향 모델 축적 + 스트리밍 딥링크.
Effort: L. Risk: Med. Spotify API third-party 제한 강화 추세 주의.

### Approach C: "AI 대화형"
LLM + YouTube Data API로 대화형 추천. 챗 인터페이스.
Effort: S-M. Risk: Low. 스와이프 게임감 없음.

## Recommended Approach

**Approach A.** 핵심 가설(스와이프 탐색이 실제로 즐거운 경험인가?)을 이번 주 안에 검증.
A가 즐거우면 B로 키우고, 별로면 방향 조정. 추측으로 B 바로 가는 건 리스크.

**Tech Stack:**
- Frontend: Next.js + Tailwind (또는 빠른 시작은 vanilla HTML/CSS/JS)
- 추천 API: Last.fm Similar Tracks API (무료, API key 즉시 발급)
  - 대안: Spotify Recommendations API (OAuth 필요하지만 추천 품질 높음)
- 배포: Vercel (무료, 5분 배포)
- 취향 저장: localStorage (초기), 추후 Supabase

**핵심 화면 (3개):**
1. 온보딩: "좋아하는 노래 5곡 입력해줘" (아티스트명 + 트랙명)
2. 탐색: 카드 스와이프 UI — 노래 제목, 아티스트, 장르 표시. 오른쪽=좋아요, 왼쪽=패스.
   하단에 장르이탈 버튼 (🎲 "완전 다른 거") — 별도 상태로 관리
3. (나중에) 저장된 좋아요 목록 + 스트리밍 링크

## Open Questions

- Last.fm vs Spotify Recommendations API 중 어느 쪽이 추천 품질이 더 나은지 실험 필요.
- "완전 다른 거" 버튼의 장르 다양성 정도를 어떻게 정의할지.
- 재생 종료 후 자동 복귀는 기술적으로 불가 (모바일 딥링크 한계) — 유저가 수동 복귀.

## Technical Spikes (만들기 전에 먼저 확인할 것들)

1. **Last.fm API 저명도 트랙 테스트** — 내가 실제로 듣는 노래 중 덜 유명한 곡을
   Last.fm API에 넣어봐. 결과가 비면 Spotify API로 전환 또는 둘 다 fallback으로 연결.

2. **메타데이터 없는 경우 graceful degradation** — API가 장르/아티스트 정보 없이
   반환할 경우: 카드에 "장르 정보 없음" 표시, 아티스트명은 "Unknown Artist" 기본값.
   테스트는 목업 데이터로 먼저 진행.

3. **localStorage 스키마 (초기 MVP)**
   ```
   savedLikes: [{ trackId, trackName, artist, rating, timestamp }]
   passedTracks: [trackId]  // 다시 보여주지 않기 위해
   ```
   다기기 동기화는 MVP 범위 외.

4. **장르이탈 버튼 동작 명세**
   - 누르면 다음 5장의 카드가 "다른 장르 추천 소스"에서 옴
   - 5장 후 자동으로 메인 추천 체인으로 복귀
   - 메인 취향 모델(좋아요/싫어요 기록)에 영향 없음
   - 버튼 활성 상태일 때 카드에 시각적 표시 (예: 🎲 아이콘)

5. **모바일 스와이프 구현**
   - 라이브러리: Hammer.js 또는 CSS transform + touch event 직접 구현
   - 테스트 환경: iPhone Safari 15+ 우선 타겟
   - 뷰포트: `<meta name="viewport" content="width=device-width, initial-scale=1">`
   - 의존성으로 추가: Hammer.js (CDN 또는 npm)

## Success Criteria

- 이번 주 내에 본인이 실제로 5일 이상 써볼 수 있는 상태로 배포.
- 추천 10곡 중 2곡 이상 "오 이거 좋다"가 나오면 추천 품질 합격.
- 스와이프 자체가 지루하지 않으면 UX 가설 검증.

## Distribution Plan

- Vercel 배포, 링크 공유로 친구 3-5명에게 써보게 함.
- 초기에는 앱스토어 필요 없음 — 모바일 브라우저 PWA로 충분.
- 피드백 받으면 Approach B(앱스토어 출시)로 확장 결정.

## Dependencies

- Last.fm API key (무료, https://www.last.fm/api/account/create)
- 또는 Spotify Developer App (https://developer.spotify.com/dashboard)

## The Assignment

이번 주 안에 Last.fm API key 발급받고, vanilla HTML 파일 하나로 씨드 트랙 입력 → 추천 리스트 받아오는 것만 만들어봐. 스와이프 UI는 그 다음. API 연결이 실제로 되는지 확인하는 게 첫 번째 진짜 과제야.

## What I noticed about how you think

- "이 앱이 사라지면 다시 음악적 깊이가 갇힐 것 같아"라고 했어. 이게 수요 근거를 설명하는 방식치고는 꽤 정확해. 마케팅 언어 없이 실제 손실을 묘사한 거야.
- 스트리밍 연동 아이디어(재생 후 자동 복귀 + 평점 요청)가 가장 창의적인 부분이었는데, 기술적 제약을 듣고 바로 "나중에 붙여도 됨"으로 전환했어. 집착 없이 실용적으로 조정하는 태도.
- 장르이탈 버튼이 "메인 알고리즘에 영향 안 줘야 해"라고 처음부터 못 박은 것 — 이건 제품 설계자처럼 생각하는 거야. 대부분은 이걸 나중에 깨달아.
